# sql/

SQL-миграции целевой Supabase-схемы (§38 `PLATFORM_TARGET_ARCHITECTURE.md`).
Выполняются **вручную** — приложение и anon-ключ не могут выполнять DDL.

## Как выполнить `002_create_platform_v2_tables.sql`

1. Открыть Supabase Dashboard связанного проекта (`https://api.rbi-q.ru`).
2. Перейти в **SQL Editor**.
3. Вставить содержимое файла целиком, нажать **Run**.
4. Убедиться, что выполнение прошло без ошибок (создаются 13 новых таблиц
   + RLS-политики).

## Если ошибка «таблица уже существует»

Не будет — все `CREATE TABLE IF NOT EXISTS`, повторный запуск безопасен.
Если ошибка на этапе `DO $$ ... $$` (политики) — вероятно, политика с тем
же именем уже создана вручную; скрипт сам делает `DROP POLICY IF EXISTS`
перед созданием, повторный запуск также безопасен.

## Откат

Раскомментировать блок `ROLLBACK` в конце `002_...sql` и выполнить —
удаляет все 13 новых таблиц. Старые таблицы не затрагиваются никогда.

## Как выполнить `003_extend_contractors_and_contracts.sql`

1. Открыть Supabase Dashboard связанного проекта (`https://api.rbi-q.ru`).
2. Перейти в **SQL Editor**.
3. Вставить содержимое файла целиком, нажать **Run**.
4. Убедиться, что выполнение прошло без ошибок (6 новых колонок у
   `contractors` + новая таблица `contracts` с RLS-политиками).

Повторный запуск безопасен — `ADD COLUMN IF NOT EXISTS`/
`CREATE TABLE IF NOT EXISTS`, политики пересоздаются через
`DROP POLICY IF EXISTS`.

Откат — раскомментировать блок `ROLLBACK` в конце `003_...sql`. Старые
таблицы и существующие 7 полей `contractors` не затрагиваются никогда.
