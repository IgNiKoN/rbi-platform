# RBI Platform — Architecture Brief

> Организационные вехи, риски, команда, права на платформу, этапы 2–6 (после Platform Foundation) — см. `_ai/PROJECT_CARD.md` (карточка проекта, бизнес-документ). Не дублируется здесь — этот файл и `PLATFORM_TARGET_ARCHITECTURE.md` только про техническую архитектуру текущего этапа (Platform Foundation).

## Target chain
```
index.html
→ js/core/app.entry.js
→ js/core/rbi-core.js
→ js/services/*.service.js
→ js/modules/modules.manifest.js
→ js/modules/<module>/manifest.js
→ js/modules/<module>/index.js
→ internal module files
```

## Rules
- manifest.js + index.js, не manifest-only.
- index.js — публичный вход модуля.
- Core не знает внутренние файлы модулей.
- Legacy временный, не финальная архитектура.
- app.js постепенно сокращается.
- index.html постепенно становится application shell.
- Модули работают через ctx/services/events.
- Не обращаться напрямую к IndexedDB, Supabase, dbPut, dbGet, STORES, triggerSync.
- sync.js, storage.js, sw.js, IndexedDB, Supabase, авторизацию трогать только отдельным блоком.
- Новые бизнес-модули не начинать до завершения platform foundation.
- **Публичная граница модуля (зафиксировано 2026-07-13, прямое указание пользователя, §29 вне очереди):** только `index.js` (или `<module>.module.js`, если это фактический entry — сверить по `js/modules/modules.manifest.js`) модуля имеет право писать `window.<Namespace> = ...`. Все файлы внутри `features/**` — чистые ES-модули (`export`/`import`), не создают собственных window-глобалов; публичный доступ к их функциям — только через entry-файл модуля, который импортирует их и сам решает, что вынести в `window` (переходно) или в registry/ctx (целевое). Правило действует со дня принятия для нового/изменяемого кода — существующие нарушения (20+ файлов на 2026-07-13) фиксируются как inventory техдолга и не блокируют текущие блоки автоматически, устраняются отдельным будущим блоком. Автоматическая проверка — `_ai/scripts/check-module-boundaries.sh` (baseline `_ai/scripts/known-boundary-debt.txt`), обязательна для блоков, трогающих `js/modules/**` — см. `agents/implementer.rule.md`/`agents/architect.rule.md`.

## Full Application Migration
Текущее приложение RBI Quality Pro полностью переезжает в RBI Platform. RBI Platform — не оболочка рядом со старым приложением, а целевая архитектура самого приложения. Ничего не остаётся постоянным legacy-блоком: каждая функция становится platform module / internal feature / service / shared / temporary proxy.

## Compact Module Policy
platform module = крупное бизнес-направление. Целевые modules: `quality`, `sk`, `construction`, `knowledge`, `settings`, `ai`, `gamification`.
`quality` = один platform module. `audit`/`history`/`tasks`/`reports`/`analytics`/`etalon`/`engineer`/`schedule`/`meetings`/`interventions` = internal features (сейчас переходно оформлены как 9 отдельных manifest/index — план консолидации: `_ai/COMPACT_MODULE_RESTRUCTURE_PLAN.md`).

## Глоссарий: бизнес-термин ↔ модуль в коде (не путать при чтении карточки проекта)
Бизнес-карточка проекта использует термины «строительный контроль»/«СК»/«МРС» неоднозначно относительно имён папок кода — сверка:
- «СК» / «ПК СК» / «строительный контроль» **в старом, переходном смысле** (импорт и анализ Excel-выгрузок из внешней системы) = модуль **`sk`** (`js/modules/sk/`). Analytics/reporting-подмодуль, не производственный контур — см. `PLATFORM_TARGET_ARCHITECTURE.md` §2.4.1.
- «Строительный контроль» / «замена МРС/Стройконтроль» **в целевом смысле** (выдача/снятие замечаний, замечания на планах, приёмка работ, шахматка/передача) = модуль **`construction`** (`js/modules/construction/`, `ConstManager`/`ConstAcceptance`/`TransferManager`). Это будущий второй production platform module (Этап 3 карточки проекта), уже частично существует в коде.
- «Модуль качества» карточки = platform module **`quality`** (internal features: `audit`/`history`/`tasks`/`reports`/`analytics`/`etalon`/`engineer`/`schedule`/`meetings`/`interventions`).
- Будущие бизнес-направления карточки (ПТО, документы, сроки, бюджет, СДО, тендерный отдел, юридический блок, клиентский сервис, гарантийное сопровождение) технически соответствуют плейсхолдер-именам §20 `PLATFORM_TARGET_ARCHITECTURE.md` (список актуализирован 2026-07-12 по сверке концепции платформы с пользователем: `safety`, `warranty`, `technical-decisions`, `contractors`, `analytics`, `schedule`, `budget`, `operation` — заменяет прежний `production/*`) — точное соответствие «бизнес-имя → папка» фиксируется в момент создания каждого модуля, не заранее (YAGNI).

## Legacy includes
Весь перечень standalone legacy JS (`ai.js`/`export.js`/`game.js`/`etalon.js`/`faq.js`/`changelog.js`/`router.js`/`views.js`/`templates.js`/`math.js`/`roles.js`/`objectDirectory.js`/`contractorDirectory.js`/`constructionManager.js`/`transferManager.js`) и все 9 `*.legacy.js`-прокси физически удалены (2026-07-07) — раздел закрыт, актуальны только:
- inline `onclick`/`onchange` в `index.html` (не разобраны, см. `INDEX_HTML_HANDLERS_MAP.md`);
- старые script-подключения в `index.html` (сокращаются по мере переноса);
- `window.*` прокси без плана удаления (актуальны — большинство модулей всё ещё общаются через `window.*`, план явного устранения не начат).

`js/app.js` (657 стр.) — отдельный статус, не приравнивается к обычным legacy standalone JS: это application shell + межмодульный global-state слой (глобальные переменные `state`/`details`/`photos`/`contractorArray`/`etalonActsArray`/`userTemplates`/`reportsArray`/`currentTemplateKey`/`currentChecklist`/`appSettings`, `DOMContentLoaded`-инициализация, `restoreSession()`), требующий отдельного крупного архитектурного решения (куда переезжает общий state), а не разбора в рамках обычного legacy-блока.

## Tracked deferred items (не забыть при планировании следующих блоков)
Зафиксированы в `PLATFORM_TARGET_ARCHITECTURE.md`, встроены в последовательность §29 — архитектор перечитывает §12/§29 каждые ~5 шагов (см. `agents/architect.rule.md`), но эти пункты также держим здесь, в ежедневном контексте, чтобы не пропустить момент их старта:
- **Роли по организационной структуре** (§23) — вместе с блоком Auth Gate + User Context или сразу после него.
- **Кросс-модульный управленческий дашборд** (§37.4) — после Platform Entry и появления 2–3 production-модулей сверх `quality`.
- **Переход на новую схему Supabase-таблиц** (§38, единый стандарт полей §18.1) — Блок 1 (проектирование схемы/маппинга, чистое документирование) можно делать параллельно с текущим Platform Foundation, не дожидаясь его завершения. Блоки 2–6 (создание таблиц, переключение записи — порядок СК → Стройконтроль → Качество, миграция истории, вывод старых таблиц) — только после завершения Foundation; Блок 4 — критичный шаг с отдельным подтверждением на каждый модуль. Старые таблицы никогда не удаляются автоматически. **Механизм миграции истории (§38 Блок 5, согласован с пользователем 2026-07-12):** UI внутри `settings`, видимый только роли `admin` («Миграция данных»), выполняется в браузере через существующий Supabase-клиент (не серверная функция — service_role/CLI недоступны в проекте), порциями с прогресс-баром по каждой из 13 таблиц; записи без завершённой нормализации подрядчика/локации (`canonical_key`=`pending`) пропускаются и подхватываются повторным запуском после ручной нормализации через уже существующий механизм очереди (`ContractorDirectory`/`ObjectDirectory`), не блокируют и не требуют 100% чистоты перед первым запуском. Для каждого модуля Блок 5 запускается только после того, как Блок 4 того же модуля закрыт и подтверждён — не одним совмещённым шагом.
- **Единая карточка подрядчика** (Веха 7 карточки проекта — данные качества и строительного контроля в одной карточке) — логическое следствие единой модели данных (§18) и единого справочника подрядчиков (`ContractorDirectory`); реалистична только после появления модуля `construction` как полноценного второго production-модуля (Этап 3 карточки) и/или Блока 4 §38 (переключение записи качества и СК на единую схему с `contractorId`). Не начинать раньше этих предпосылок.
- **Аналитика «узел × подрядчик»** (§18.2, зафиксировано 2026-07-12 по запросу пользователя) — отслеживание поведения технического узла (`Node`) у конкретного подрядчика: дефекты по узлу, повторяемость, применимость, рекомендации. Требует физического создания `Node`/`NodeApplication` (сейчас не существуют, только описаны в §18) и завершения Блока 4 §38 (`contractorId` вместо свободной строки у дефектов `construction`). Не начинать раньше этих предпосылок.
- **Расширенное видение жизненного цикла объекта** (§18.3, зафиксировано 2026-07-12 по расширенному описанию платформы пользователя) — цепочка «объект → вид работ → узел → подрядчик → проверка → дефект → предписание → устранение → гарантийное обращение → аналитика → корректировка стандартов», детальный набор сквозных ID (включая новые `zoneId`/`workTypeId`/`warrantyCaseId`/`eventId`), принцип истории юридически значимых событий для сущностей с юридическим весом. Чисто дополнение горизонта планирования — не выполняется сейчас. **Открытый вопрос без решения:** предложенный «Entity Envelope» расходится по именам/составу полей с уже действующим §18.1 (стандарт новых Supabase-таблиц) — не сводить в один стандарт и не менять §18.1/`sql/002_create_platform_v2_tables.sql` без отдельного явного решения пользователя.
- **Планы этажей как общая сущность + консолидация Engineer/User** (§18.4, зафиксировано 2026-07-12 по сверке концепции платформы с пользователем) — план этажа (сейчас только в `construction`) в перспективе общая платформенная сущность над `LocationNode`, не начинать раньше появления второго реального потребителя кроме `construction`. `engineerId` признан не отдельной сущностью, а ролью `userId`/`User` — сведение свободной строки `engineerName` к `userId` — часть будущего блока §23 (роли по оргструктуре), не отдельная задача.
- **Карточки будущих модулей-заглушек на экране выбора модуля** (§37.1) — больше не отдельный блок, поглощены Блоком 2 App Shell ниже (welcome-экран включает все карточки сразу).
- **Целевая структура разделов внутри `settings`** (§37.2, дополнение 2026-07-12) — настройки качества/стройконтроля как подразделы единого раздела «Настройки», а не отдельные экраны каждого модуля. Зафиксировано как будущее состояние, не начинать реструктуризацию `settings.render.js` без конкретной задачи, которая в ней реально нуждается (YAGNI).
- **Перенос статичной разметки `quality`/`app-modals` в JS-рендер модулей** (зафиксировано 2026-07-13, прямой запрос пользователя в чате). Уточнение целевой модели: `index.html` рисует только App Shell (sidebar/header/company block/точки монтирования `#app-content`/`#app-modals`); каждый модуль сам генерирует свою разметку через `<module>.render.js` и монтирует её в эти точки — единый визуальный стиль при этом управляется платформой централизованно (`css/platform.tokens.css`), модули не привносят свой независимый стиль (согласуется с §24). Текущее состояние (после App Shell Блок 1 + зона `app-modals` Блок 1/2): контейнеры `#app-content`/`#app-modals` созданы и физически содержат нужную разметку, но сама разметка `quality` (5 features) и 31 модального/overlay-блока остаётся статичным HTML в `index.html`, не генерируется `*.render.js`. Это крупный и рискованный блок (переписать генерацию сотен строк HTML для каждой фичи `quality`, не сломав ни одной существующей функции) — не начинать без отдельного явного решения пользователя о приоритете и без завершения текущей стабилизации/изоляции платформы. Также требует полноценного sidebar-навигации (сейчас есть только welcome-экран/company block/переключатель режима, не настоящий sidebar с иконками модулей).
