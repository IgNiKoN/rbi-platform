**КРИТИЧНЫЙ шаг** (трогает `js/services/sync/*`, вопрос 2 — миграция/сверка данных). Не переходить к реализации без подтверждения пользователем решений по открытым вопросам §8.

## 2. Диагноз (подтверждено чтением кода)

### 2.1. Причина «плашка заявки есть не у всех» — баг НЕ пофиксен в предыдущем блоке

В прошлом блоке (см. историю ниже, `_ai/CURRENT_STEP.md`) единый fallback между `assigned_projects`(колонка) и `settings.assignedProjects`(JSON) был исправлен в `permission.service.js:getAssignedProjects()` и в pull `sync-engine.core.js`. **Но не в панели «Команда»** — там своя, отдельная, старая копия той же логики:

```1798:1801:js/modules/quality/features/gamification/game.actions.js
        const projectsArray = Array.isArray(user.assigned_projects)
          ? user.assigned_projects
          : Array.isArray(user.settings?.assignedProjects)
            ? user.settings.assignedProjects
            : [];
```

Здесь тот же баг: если `user.assigned_projects` — пустой массив `[]` (а не `null`/`undefined`), `Array.isArray([])===true`, и код никогда не смотрит в `settings.assignedProjects`, даже если там реально лежат назначенные объекты (расхождение между двумя полями — предмет прошлого блока, само расхождение НЕ устранено полностью, потому что писать могли админ через `gameSaveUserAccess` в старых записях, до фикса, либо какой-то путь всё ещё пишет только одно поле). Так админ видит «объект не закреплён», хотя по факту у инженера есть данные в `settings.assignedProjects`.

Второй экземпляр той же логики — при построении payload `requestedProjects` (не про сами объекты, но по аналогии нужно свериться).

### 2.2. Причина «у одних плашка заявки есть, у других нет, хотя должна быть у всех, кто её создал»

Заявки (`requestedProjects`) создаются 3 разными путями, каждый со своими условиями/фильтрами:

1. `pushObjectRequestToCloud` (`sync-connection.actions.js:57-92`) — используется `addAssignedProject` (пользователь добавляет объект у себя в настройках синка) — пишет в `settings.requestedProjects` через `upsert`.
2. `audit.actions.js:430-442` — при создании осмотра инженер вручную вбивает новый объект (не выбирает из справочника) — отдельный, самостоятельный код, ПОЧТИ идентичный п.1, но без дедупликации по `canonical_key`, без учёта `request_type`, и, что важнее — **не использует `pushObjectRequestToCloud`**, а пишет прямым `update()` в `settings` (не через `writeUserProjectAssignment`, появившийся в прошлом блоке). Тут же есть скрытая опечатка риска: `.then()` без `.catch()` — ошибка сети/RLS проглатывается молча, заявка может не долететь, а пользователь увидит "отправлено".
3. `construction/features/admin.js:280-300` — аналогичный третий independent код-путь (та же логика, что и в п.2, для модуля "Стройконтроль"), тоже мимо `pushObjectRequestToCloud`/`writeUserProjectAssignment`.

Дальше в панели «Команда» (`gameLoadRoles`) заявки фильтруются:

```2146:2151:js/modules/quality/features/gamification/game.actions.js
      let requestedProjects = Array.isArray(currentSettings.requestedProjects)
        ? currentSettings.requestedProjects.filter(r =>
          r.source !== 'sk_import' &&
          r.request_type !== 'directory'
        )
        : [];
```

Заявки с `request_type === 'directory'` (создаются в п.2/п.3 выше — audit.actions.js/admin.js ставят `request_type: 'directory'` через локальный код, аналогичный `addAssignedProject`, где `requestStatus` не matched → в дальнейшем при отправке `pushObjectRequestToCloud` получают `request_type: 'directory'` при вызове с этим типом) **отфильтровываются и не показываются админу в панели «Команда» вообще** — они относятся к другому фильтру (заявка на пополнение справочника, обрабатывается в `object-directory.service.js:loadRequests` отдельным блоком «Заявки из ПК СК»/справочника). Т.е. если инженер вбил новый объект вручную при создании осмотра (`audit.actions.js`), заявка на **привязку к пользователю** там реально не создаётся — создаётся только заявка на **пополнение справочника** (если объект не нашёлся). Привязка конкретного инженера к этому объекту не запрашивается вовсе в этом пути. Из-за этого у части инженеров, вбивавших объект вручную через шапку осмотра, нет видимой заявки в панели «Команда», хотя они явно "запросили" объект.

**Итог:** админ видит несогласованную картину, потому что (а) само чтение закреплённых объектов содержит старый баг fallback, не исправленный в панели «Команда»; (б) создание заявки на привязку реализовано в 3 разных местах с разной логикой, 2 из 3 не создают видимую в панели «Команда» заявку на привязку вообще, только заявку на справочник.

### 2.3. Офлайн → онлайн новая учётка → сверка справочников (текущее поведение, подтверждено чтением)

Текущий процесс подключения (`initCloudConnection`, `sync-connection.actions.js:247-345`):
- Создаёт профиль `guest`/`pending` в Supabase, **не переносит и не анализирует локальные офлайн-данные вообще** — только резервирует identity (`auth_user_id`/`pin_hash`).
- После подтверждения админом (`gameSaveUserAccess`/`resolveRequest`) и следующего pull — включается push локальных данных (`sync-engine.core.js`, `actuallyPushedChecks`/`actuallyPushedTasks`/`actuallyPushedProfiles`), но это push уже существующих записей (осмотры/задачи), **не сверка названий объектов** из этих локальных записей со справочником.
- Названия объектов в локальных офлайн-записях (`projectName`/`project_canonical_key`) **не проходят автоматическую нормализацию/сверку со справочником при первом подключении** — единственный путь нормализации, `ObjectDirectory.normalizeProjectName()`, вызывается только точечно (при вводе через `addAssignedProject`/шапку осмотра в реальном времени), не пакетно по всей истории офлайн-записей.
- Соответственно, если офлайн-инженер месяц вводил "Объект №5" (не по справочнику), а в справочнике он записан как "ЖК Северный, корпус 5" — после подключения к облаку и push этих записей они попадут в облако с "сырым" именем, без предложения админу на сопоставление/создание. Нет отдельного UI-экрана «Сверка объектов нового пользователя» для админа.
- Частичный механизм есть только для будущих (не прошлых) объектов через `object_normalization_queue` (`resolveDirectoryRequest`) — но туда попадают только явные заявки на **новый** объект в справочнике, создаваемые в реальном времени при вводе, не retroactive-сканирование истории.

## 1. Диагноз

### 1.1. История пустая после полного сброса+синка, появляется только после F5

Событие `sync:completed` (задокументировано в `PLATFORM_TARGET_ARCHITECTURE.md` §5, на него подписаны 12 модулей — `history`/`audit`/`analytics`/`tasks`/`sk`/`etalon`/`meetings`/`schedule`/`reports`/`engineer`/`game`/`construction`) **нигде в коде не эмитилось**. Проверены все возможные точки эмита (`RBI.events.emit('sync:completed', …)`, `document.dispatchEvent(new CustomEvent('sync:completed'))`) по всему `js/**` — ни одной. `triggerSync()` (`sync-engine.core.js`) после завершения только выставляет `window.syncDirtyFlags.history/tasks/analytics/... = true` и точечно обновляет некоторые вкладки (аналитика/задачи/справочник практик-эталонов) напрямую вызовами рендер-функций, но НЕ Историю — и не эмитит событие, на которое подписаны модули. Единственный путь, которым `HistoryState.allRecords` реально наполнялось после полного сброса — перезагрузка страницы, которая заново вызывает `session.service.js:restoreSession()` → `dbGetAll(STORES.HISTORY)`.

### 1.2. Глобальные фильтры (мультифильтр объект/подрядчик/инспектор) в истории не работают

Split-brain state: `applyMultiFilter()` (`multi-filter.js`) при нажатии «Применить» пишет выбор в `window.activeMultiFilters.history.project/contractor/inspector`. `HistoryRender.render()` (`history.render.js`) читал fProj/fContr/fInsp из **другого** объекта — `HistoryState.filters.project/contractor/inspector`, который никогда не заполняется этими тремя ключами (там только `period`/`searchText`, дефолт даже не массив, а строка `''`). Та же проблема была найдена и обойдена ранее в `analytics.actions.js`/`analytics.render.js` (`_historyFilters()`/`_analyticsFilters()` — комментарий там прямо описывает баг «фильтр визуально выбран, но данные показываются нефильтрованными»), но `history.render.js` этот обходной путь не использовал.

### 1.3. UX мультифильтра неудобен

Раньше вся строка была одним `<label>`, оборачивающим `<input type="checkbox">` + текст — клик в любое место строки просто переключал чекбокс (множественный выбор), реальное применение требовало отдельного нажатия «Применить». Нужно: клик по тексту — сразу выбрать только это значение и закрыть модалку (быстрый однозначный выбор); клик по чекбоксу — обычное переключение мультивыбора без закрытия (для набора нескольких значений через «Применить»).
