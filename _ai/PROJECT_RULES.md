# RBI Platform — Общие правила проекта

Действуют для всех ролей (Архитектор, Исполнитель, Тестер, Документер) всегда.

---

## Экономия контекста

- Не сканировать директории и не читать файлы проекта целиком «на всякий случай». Читать только то, что явно нужно для текущей задачи.
- Источник истины по текущей задаче — файлы в `_ai/` (`current_plan.md`, `CURRENT_STEP.md`, `PLATFORM_TARGET_ARCHITECTURE.md`, `SERVICES_API.md`, `MODULE_CROSS_REFERENCE_MAP.md`), не весь код проекта.
- Не читать реализацию сервиса, если существует `SERVICES_API.md` — использовать справочник вместо чтения кода.
- Вопрос «что делает файл X» / «кто из модуля A вызывает что из модуля B напрямую (bare `window.*`)» — сначала смотреть в `_ai/MODULE_CROSS_REFERENCE_MAP.md`, не через `Grep`/чтение файлов заново. Grep/Read по коду — только для (а) точечной проверки, что найденная в карте строка не устарела, (б) деталей, которых карта не даёт (сама реализация функции).
- Массовый параллельный аудит всего проекта (несколько subagent'ов, сканирующих весь `js/**`) — разовая операция для построения/капитального пересмотра карты, не повторяется каждый цикл. Обычный цикл работает точечно: план/шаг/отчёт → 1–3 целевых файла, не десятки.
- Не дублировать в чате то, что уже записано в файл — после записи выводить только краткое резюме (что сделано, куда записано).

---

## Размер блока работы

- Один блок = законченный, ощутимый кусок (весь модуль целиком, один паттерн применённый ко всем похожим файлам сразу, весь этап архитектуры целиком, вся фича целиком — с UI, состоянием и подключением, а не только один из этих слоёв).
- Запрещено дробить на «добавить одну функцию», «заменить 2–3 строки», «переименовать одну переменную», «добавить один DOM-узел + пару строк-обёрток + один вызов» как отдельные блоки — такие изменения объединяются в один блок со связанной по смыслу работой, пока не наберётся реальный ощутимый кусок.
- Если задача кажется мельче, чем чтение контекста для неё — она слишком мелкая, объединить со следующим логическим шагом.
- **Количественный ориентир (не жёсткий норматив, но сигнал тревоги):** если предполагаемый блок правит меньше ~40–60 строк суммарно по коду (без учёта `.md`) и не закрывает пункт чек-листа §29 полностью — это сигнал, что блок нужно расширить, а не фиксировать как есть. Планировать сразу на 1 логический шаг вперёд не достаточно — нужно закладывать в блок весь объём работы, который реально относится к текущему пункту чек-листа и может быть сделан безопасно за один раз.
- **Анти-пример (слишком мелкий блок):** «добавить один пустой `<div>` в шапку + 2 метода-обёртки в сервисе + один вызов рендера из точки инициализации» — это не блок, а фрагмент блока. Правильный масштаб для той же задачи (пример App Shell/Header): сразу весь user-facing блок шапки, который реально нужен (имя+роль+статус синхронизации+переключение, если это одна связанная зона UI), либо перенос сразу нескольких точек одного паттерна, а не одна точка.
- Перед тем как фиксировать план, Архитектор должен явно спросить себя: «эта работа — это весь ощутимый шаг, который можно сделать сейчас безопасно, или я специально порезал более крупную задачу на кусочки, чтобы не иметь дело с её полным объёмом сразу?» Если ответ — второе, план нужно расширить до реального объёма шага.
- **Чисто документационные находки не образуют отдельный блок.** Если Архитектор при чтении карт (`_ai/*.md`) замечает устаревшую строку, неточный путь файла, опечатку, несмещённый номер строки и т.п. — без единого изменения кода/поведения приложения — это НЕ отдельный блок и НЕ повод запускать цикл Архитектор→Исполнитель→Тестер→Документер. Архитектор чинит это сам, напрямую, в рамках того же ответа, без `current_plan.md`/`LATEST_ARCHITECT_PLAN.md` и без Исполнителя. Если находок много и разрозненных — можно собрать их в один такой прямой проход, но не превращать в план.
- **Минимальный порог блока Архитектора:** план в `current_plan.md` создаётся только если блок содержит хотя бы одно из — перенос/создание/удаление файла кода, изменение подключения/регистрации модуля, изменение поведения приложения. Если предполагаемая работа — правка только `.md`-файлов без единого изменения кода, это не блок Архитектора.
- Каждый блок должен быть достаточно крупным, чтобы оправдать полный цикл ролей: ориентир — не менее одного целого файла кода целиком, или одной сквозной находки, применённой сразу везде, или одной полноценной user-facing фичи целиком (не одного её слоя/точки), а не одна строка/один DOM-узел/одна ссылка в документации.
- Если следующий незакрытый пункт §29 сам по себе крупный (App Shell, Auth Gate и т.п.) — не растягивать его на серию мелких «шаг 1, шаг 2, шаг 3...» с крошечным диффом в каждом. Один такой блок должен закрывать столько реального объёма пункта, сколько можно сделать безопасно за раз (в идеале — весь пункт целиком, если это выполнимо без нарушения Definition of Done); дробление на подшаги допустимо только если пункт объективно требует нескольких независимо проверяемых критичных решений (например, разные файлы высокого риска), а не потому что так проще спланировать.

## YAGNI — не создавать «на будущее»

- Не создавать файл, сервис, таблицу или поле, если ни одна текущая задача их не использует прямо сейчас.
- Каждый новый файл должен отвечать на вопрос «какую реальную, существующую сейчас проблему он решает» — если ответа нет, не создавать.
- Не проектировать multi-tenant, темизацию под чужие бренды, Integration/AI Platform, workflow-engine и подобную инфраструктуру заранее — добавлять по факту первой реальной задачи, которая их требует.

---

## Что нельзя трогать без явного указания в текущем плане

- Файлы вне списка «что изменить» / «что создать» из `current_plan.md`.
- Не удалять старые/legacy-файлы без явного разрешения в плане — strangler fig: новое строится рядом, старое убирается отдельным подтверждённым шагом.
- `storage.js`, `sync.js`, `sw.js` не защищены постоянным запретом — это не «неприкасаемые» файлы навсегда, а файлы с высоким риском. Их можно и нужно менять, когда это реальный следующий шаг рефакторинга (они тоже должны переехать в целевую архитектуру), но только отдельным явным критичным блоком (см. «Критичные шаги» ниже) — не «заодно» внутри блока с другой целью.

## Критичные шаги (требуют расширенной проверки)

**Критичные** (не переходить к следующему шагу без явного подтверждения от пользователя после проверки):
App Shell, Auth Gate, Module Loader, изменения синхронизации, изменения `storage.js`/`sync.js`/`sw.js`, миграция данных.

**Для критичных шагов обязательна реальная проверка, не только `node --check`/grep:**
- Headless-браузер (Playwright/Chromium) — полная загрузка приложения, 0 `console.error`/`pageerror`/`requestfailed`, `page.reload()`.
- Для блоков, затрагивающих `config.js`/`storage.js`/`sync.js` — обязательна **реальная проверка облачной синхронизации**: тестовая запись с маркером `__SMOKE_TEST__`, тестовый доступ из `_ai/TEST_CREDENTIALS.md`, синк → подтверждённое появление записи в Supabase → удаление тестовых данных из облака и локально.
- Без этой реальной проверки критичный шаг не считается закрытым — план/отчёт не может быть принят только на основании статического анализа кода.

**Обычные** — стандартный цикл без дополнительных подтверждений, но и в обычных шагах, если они реально касаются рендера/данных, ожидается минимум headless-браузерный smoke-check, а не только чтение кода.

---

## Главная цель платформы

Строим платформу, на которой **текущее рабочее приложение продолжает работать точно так же, как сейчас**, но уже подключено к платформе (ядро / сервисы / registry), а не переписываем приложение заново. Каждый блок должен приближать именно к этому состоянию, а не создавать архитектуру отдельно от рабочего приложения.

---

## Документирование

- Архитектор пишет план в `current_plan.md` (полностью заменяя предыдущее содержимое — история уже в `CURRENT_STEP.md`).
- Исполнитель после успешной проверки сам дописывает результат в `CURRENT_STEP.md` (не переписывая историю) и, если менялся публичный интерфейс сервиса — в `SERVICES_API.md`.
- Если что-то в плане неясно для выполнения — не додумывать и не читать весь проект в поисках ответа, сообщить об этом и остановиться.

---

## Целевая архитектура (кратко)

`index.html → bootstrap.js → Core → Services → Modules`

Core — маленький, без бизнес-логики. Модули общаются только через события (EventBus) и сервисы (Context), не напрямую друг с другом. Библиотеки (Chart.js, html2pdf, SheetJS и т.д.) вызываются модулями напрямую, без обёртки-реестра.

Полное описание — в `PLATFORM_TARGET_ARCHITECTURE.md`, сверяться с ним при любом архитектурном решении.
