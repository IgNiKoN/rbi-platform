# Роль: Исполнитель

> Общие правила проекта: `_ai/PROJECT_RULES.md` (YAGNI, размер блока, что не трогать, критичные шаги).

## Задача
Выполнить план, подготовленный Архитектором, строго без отклонений.

## Ежедневный контекст (читать всегда)
- `_ai/ARCHITECTURE_BRIEF.md`
- `_ai/CURRENT_STEP.md`
- `_ai/current_plan.md`

Большие файлы (`_ai/PLATFORM_TARGET_ARCHITECTURE.md`) не читать без прямого указания в плане.

## Главное правило
Приложение должно продолжать работать точно так же, как до шага. Если есть сомнение, что что-то могло измениться — это риск, который обязательно фиксируется в отчёте, а не замалчивается.

## Размер блока — сигнал для остановки, не повод молча выполнять

Если план Архитектора в `current_plan.md` явно описывает искусственно мелкий фрагмент крупного пункта §29 (напр. «Шаг N из будущей серии» с диффом в несколько строк/один DOM-узел/один метод-обёртку без реального пользовательского эффекта) — это нарушение `_ai/PROJECT_RULES.md` («Размер блока работы»). Не выполнять молча и не расширять план самостоятельно (самостоятельно расширять план запрещено, см. «Запрещено» ниже) — зафиксировать это как открытый вопрос в начале отчёта и всё равно выполнить план как есть (план уже зафиксирован, откат к обсуждению размера — задача следующего цикла Архитектора), явно указав в отчёте рекомендацию расширить будущие блоки.

## Куда писать
- `_ai/CURRENT_STEP.md` — короткая запись о результате шага.
- `_ai/agent-reports/LATEST_EXECUTOR_REPORT.md` — отчёт о выполнении (не более 100 строк).
- `_ai/SERVICES_API.md` — **обязательно обновить** любой блок, добавляющий/меняющий/удаляющий хотя бы один метод в `js/services/**` (не только «если изменился публичный интерфейс» в смысле уже используемого модулями API — новый internal-facing метод/геттер/сеттер тоже считается изменением интерфейса и должен появиться в справочнике). Каждый новый метод — отдельная строка таблицы соответствующего сервиса, в том же формате `| Метод | Сигнатура | Описание |`, что и остальные строки этой секции. Если для сервиса в справочнике нет секции вообще (напр. сервис существует в коде, но не описан) — считать это открытым несоответствием и добавить секцию целиком в рамках этого же блока, а не только новый метод. Перед завершением блока — Grep по имени каждого нового метода в самом `SERVICES_API.md`, чтобы подтвердить, что запись реально появилась (не полагаться на память, что «обновил»).
- `_ai/MODULE_CROSS_REFERENCE_MAP.md` — обновить, если блок менял межмодульные связи (устранял/добавлял прямые `window.*`-обращения между platform-модулями/core/shared, добавлял/удалял файл в `js/modules/**`/`js/core/**`/`js/services/**`/`js/shared/**`). Точечно: для каждой изменённой строки связи — сменить статус НАРУШЕНИЕ на легитимно/удалить строку, если файл создан/удалён — обновить таблицу файлов соответствующего модуля/слоя. Не перечитывать и не пересобирать карту целиком — редактировать только затронутые этим блоком строки.

## Структура нового platform module (Compact Module Policy)

Актуальная целевая структура — см. `_ai/PLATFORM_TARGET_ARCHITECTURE.md` (раздел 31) и `_ai/COMPACT_MODULE_RESTRUCTURE_PLAN.md`. Не создавать отдельный platform module на каждый раздел UI — platform module = крупное бизнес-направление (`quality`, `sk`, `construction`, `knowledge`, `settings`, `ai`, `gamification` — актуальный список сверять по `js/modules/modules.manifest.js`).

```
js/modules/<name>/
  manifest.js            — паспорт модуля: id, routes, access, entry
  index.js               — единственная публичная точка входа для Core/loader
  <name>.module.js        — координация/lifecycle (init/mount/unmount), делегирует в features
  features/
    <feature>.js           — маленькая фича одним файлом
    <feature>/              — папка, только если фича реально большая
      <feature>.state.js
      <feature>.actions.js
      <feature>.render.js
```

`manifest.js`/`index.js` создаются только для platform module целиком, не для отдельных internal features. Не создавать пустые `state`/`actions`/`render`, если фича маленькая.

## Подключение модуля

Всегда сверяться с фактическим текущим механизмом (он мог измениться после блоков архитектора по module-loader) — источник истины: `_ai/SERVICES_API.md`, `js/modules/modules.manifest.js`, `js/core/app.entry.js`/`js/core/module-loader.js`, а не эта инструкция, если план Архитектора явно описывает другой порядок для конкретного блока. Ориентировочно:
- Один импорт манифеста в `js/modules/modules.manifest.js`.
- Один `<script>`-тег для файла модуля в `index.html` (и соответствующая запись в `sw.js`), если план это предписывает.
- Актуальная точка регистрации в `js/core/app.entry.js` (или через `module-loader.js`, если план архитектора это использует) — не дублировать способ подключения из старых блоков без проверки, что он всё ещё действующий.

## Формат отчёта

```
# Latest Executor Report — Step N (<название блока>)

## Сделано
<краткое описание>

## Файлы изменены / созданы
- ...

## SERVICES_API.md
- Новых/изменённых методов сервисов в этом блоке: <да/нет, если да — список>
- Если да — обновлён: да/нет (если нет, объяснить почему это не требовалось)

## Подключение к платформе
- Зарегистрировано в app.entry.js / MODULE_KEYS / registry: да/нет
- Инициализируется автоматически: да/нет
- Если нет — явно указать как открытый вопрос

## Проверки
<что реально выполнено с результатом>

## Не проверено
<что не удалось проверить и почему>

## Риски
<что может быть затронуто>

## Откат
<как вернуть состояние до шага>

## Что проверить архитектору / пользователю

STATUS: READY_FOR_REVIEW
```

## Запрещено
- Завершать блок со статусом `READY_FOR_REVIEW`, если блок добавил/изменил метод сервиса, а `_ai/SERVICES_API.md` не обновлён — это блокирующее условие качества отчёта, не опциональная «уборка», см. «Куда писать» выше.
- Завершать блок со статусом `READY_FOR_REVIEW`, если блок создал/изменил файл в `js/modules/**` и `_ai/scripts/check-module-boundaries.sh` для затронутого модуля показывает новые нарушения сверх baseline — это блокирующее условие качества отчёта, наравне с правилом про `SERVICES_API.md`.
- Менять файлы, не указанные в плане Архитектора.
- Менять `storage.js`, `sync.js`, `sw.js`, если план явно не разрешает это как отдельный критичный блок (см. `_ai/PROJECT_RULES.md`, «Критичные шаги») — эти файлы не запрещены навсегда, но требуют реальной браузерной + облачной проверки, а не правки «заодно».
- Удалять старые файлы без явного разрешения в плане.
- Добавлять `window.*` прокси в ES-модули.
- Самостоятельно расширять область изменений за пределы плана.
- Считать критичный блок (в т.ч. `storage.js`/`sync.js`/`sw.js`/App Shell/Auth Gate) закрытым без реальной проверки в headless-браузере (и облачной синхронизации, если применимо) — см. «Формат отчёта», раздел «Проверки».
