## 1. Что такое платформа

**RBI Platform Foundation** — это не новое приложение и не переписывание с нуля.

Это следующая стадия развития текущего рабочего приложения **RBI Quality Pro**: тот же продукт для управления качеством, стройконтролем, базой знаний, отчётами, задачами и AI на строительных проектах, но с платформенным фундаментом, который позволяет:

- сохранить всю текущую работоспособность (IndexedDB, Supabase sync, offline-first, PWA, историю проверок, аналитику, отчёты, задачи, СК);
- подключать текущие разделы к платформе как модули;
- постепенно разбирать `app.js` и legacy-файлы на управляемые части;
- устранять глобальные обращения (`window.*`, `contractorArray`, `appSettings`, `state`);
- в будущем добавлять новые бизнес-процессы (production, warranty, contractors, …) без переписывания ядра.

**Целевая формула:**

```
RBI Platform Foundation =
    уже созданный Platform Core (window.RBI: events, registry, createContext)
  + уже созданный service layer (js/services/*.service.js)
  + уже созданные ES-модули текущих разделов
  + app.entry.js как единая точка инициализации
  + ctx-контракт (createContext + bindCtx)
  + legacy wrappers как временный миграционный слой (НЕ финал)
  + дальнейшая декомпозиция app.js (~8 390 строк)
  + минимальная Company (config/context, без multi-tenant engine)
  + 15 manifest.js (ВЫПОЛНЕНО) + 15 index.js (следующий блок)
  + modules.manifest.js + module-loader.js (ВЫПОЛНЕНО)
  + App Shell (будущий крупный шаг)
  + центральный стиль (css/platform.*.css)
  + сохранение текущего offline-first и sync
  + план полного устранения технического долга (не маскировка через wrapper)
```

**Целевая цепочка запуска:**

```
index.html
→ js/core/bootstrap.js       ← App Shell: глобальный state/appSettings, DOMContentLoaded-init (заменил app.js, Шаг 2)
→ js/core/app.entry.js
→ js/core/rbi-core.js
→ js/services/*.service.js
→ js/modules/modules.manifest.js
→ js/modules/<module>/manifest.js
→ js/modules/<module>/index.js      ← публичный контракт модуля
→ <module>.module.js / .state.js / .actions.js / .render.js
```

`index.js` — единственная публичная точка входа модуля. Core, app.entry.js и module-loader работают только через `manifest.js` → `index.js`. Внутренние файлы (`*.module.js`, `*.state.js`, `*.actions.js`, `*.render.js`) скрыты за этим контрактом.

Платформа состоит из четырёх слоёв:

| Слой | Назначение |
|------|-----------|
| **Core** | Запуск, роутинг, EventBus, регистрация модулей |
| **Services** | Инфраструктурные сервисы: хранилище, синхронизация, права, файлы |
| **Master Data** | Единый источник справочников (объекты, подрядчики, шаблоны и др.) |
| **Feature Modules** | Независимые бизнес-процессы (качество, СК, задачи, аналитика, …) |

**Ключевой принцип:** модули не зависят друг от друга — взаимодействуют только через сервисы и события (EventBus). Никаких прямых вызовов между модулями.

**Важно:** это документ развития существующего плана, а не новый архитектурный план. Разделы 1–12 описывают исходную целевую модель (актуальны и сохранены). Разделы 13+ (ниже) — актуализация с учётом фактического состояния рефакторинга и цели RBI Platform Foundation.

## 2. Архитектурные слои

### 2.1 Core (Ядро) — `js/core/`

Отвечает за:
- Запуск приложения (`app.entry.js`)
- Клиентский роутинг (SPA-навигация)
- Регистрацию и инициализацию модулей
- EventBus (публикация/подписка на события)
- Глобальный контекст `ctx`, который передаётся в каждый модуль

**Не содержит бизнес-логики.** Ядро не знает о конкретных модулях — оно только управляет их жизненным циклом.

```
js/core/
└── rbi-core.js      — EventBus + ModuleRegistry + createContext
```

---

### 2.2 Services (Сервисы) — `js/services/`

Инфраструктурные сервисы, общие для всех модулей. Каждый сервис — единственная точка доступа к своей зоне ответственности.

| Сервис | Файл | Назначение |
|--------|------|-----------|
| **Storage** | `storage.service.js` | IndexedDB — единая точка доступа (обёртка над `dbPut/dbGet/dbGetAll/dbDelete`) |

| **Permissions** | `permission.service.js` | Права доступа: роли, модули, действия (обёртка над `window.RbiRoles`) |
| **Inspection** | `inspection.service.js` | CRUD для истории проверок (`STORES.HISTORY`) |
| **Report** | `report.service.js` | CRUD для отчётов (`STORES.REPORTS`) |
| **Files** | `file.service.js` | Фото, PDF, документы (`PhotoManager + rbiHydrateLocalImages`) |
| **ObjectDirectory** | `object-directory.service.js` | Объекты, этажи, квартиры (`window.ObjectDirectory`) |
| **ContractorDirectory** | `contractor-directory.service.js` | Подрядчики, виды работ (`window.ContractorDirectory`) |
| **Task** | `task.service.js` | CRUD задач, расписания, встреч (`STORES.TASKS/SCHEDULE/MEETINGS`) |

| **Knowledge** | `knowledge.service.js` | CRUD для базы знаний (`TWI/DOCS/NODES/ETALON`) |
| **Analytics** | `analytics.service.js` | Расчёт метрик (УрК, ИУрК, ИКО, CMI, Impact) |
| **Metrics** | *(целевой)* | Финансовые метрики, план/факт, отклонения |
| **Notifications** | *(целевой)* | Уведомления: Push, email, Telegram |
| **Workflow** | *(целевой)* | Движок согласований (заявка → цепочка утверждений) |
| **Audit** | *(целевой)* | Логирование пользовательских действий |
| **AI** | *(целевой)* | Вызовы AI-моделей (RAG, генерация, анализ) |

> Статус *(целевой)* — сервис не создан, является целью рефакторинга.

### 2.3 Master Data (Мастер-данные) — `js/master-data/`

Единый источник правды для всех справочников:

- Объекты строительства
- Подрядчики
- Пользователи и роли
- Этажи, квартиры, помещения
- Виды работ
- Шаблоны проверок (чек-листы)
- Эталоны

Все модули получают мастер-данные через `ctx.masterData.*` — не через глобальные переменные типа `contractorArray`, `appSettings`, `state`.

#### 2.4.1 Важное уточнение: `construction` vs `sk` — это разные модули, несмотря на похожее название

В UI и в разговоре оба раздела называют «Стройконтроль», но по назначению и по коду это два независимых platform module с разными данными и разной ролью на пути к целевой платформе:

| | `construction` (`js/modules/construction/`) | `sk` (`js/modules/quality/features/sk/`) |
|---|---|---|
| **Что это** | Собственный производственный процесс: дефекты на плане, приёмка работ, шахматка, передача квартир (`ConstManager`, `ConstAcceptance`, `TransferManager`) | Импорт и сопоставление (mapping) данных из стороннего ПК СК: записи, объёмы, маппинг подрядчиков/категорий (`SKState`: `records/volumes/contractorMap/categoryMap/mapping`) |
| **Как подключён** | Отдельный app-mode (`AppModeManager`, переключатель `#app-mode-selector`, значение `construction`) → отдельная секция роутинга `#/construction/*` | Вкладка внутри режима `quality` (`sk.manifest.js`: `menu.section: 'quality'`) — раздел анализа, не отдельный app-mode |
| **Связь со справочниками** | Частично: использует `object-directory.service.js` (объекты/корпуса/этажи) через `ConstructionState`; подрядчик — **см. риск ниже** | Собственный локальный `contractorMap`/`categoryMap` для сопоставления сырых имён из внешней системы с канонiческими |
| **Роль в дальнейшем развитии платформы** | Кандидат на полноценный второй production-модуль после стабилизации `quality` (соответствует вашему целевому описанию платформы, где «стройконтроль» — самостоятельный контур) | Остаётся analytics/reporting-подмодулем, интегрируется скорее как feature `quality`/`analytics`, а не как отдельный производственный контур |

**Обнаруженный технический риск (требует решения перед превращением `construction` в полноценный второй модуль):** в `js/construction/constructionManager.js` подрядчик у дефекта (`d.contractor`) хранится как **свободная текстовая строка** — значение `<select>`, заполненного из `ContractorDirectory.contractors[].display_name` либо (fallback) из сырых имён в `contractorArray`. Это не ссылка на каноническую сущность `Contractor`, а копия отображаемого имени. Это прямое нарушение принципа 2 целевой платформы («нельзя допускать разные версии одного и того же подрядчика в разных модулях») и должно быть зафиксировано как пункт технического долга перед тем, как `construction` начнёт масштабироваться как второй production platform module — см. раздел 16 «Что считать техническим долгом».

**Критерий приёмки роли подрядчика (`_ai/PROJECT_CARD.md`, Риск 4 — «недостаточная проработка роли подрядчика»):** прежде чем считать `construction` полноценной заменой МРС/Стройконтроль (Этап 3 карточки проекта), обязательна отдельная проверка полного цикла действий именно от роли `contractor` (не только `inspector`/`manager`): просмотр только своих замечаний (не всех), принятие замечания в работу, добавление фото устранения, отправка на проверку инженеру. Это не текущий блок Foundation — фиксируется здесь как обязательный пункт Definition of Done будущего блока «модуль `construction` как замена МРС», чтобы не быть забытым при планировании этого блока.

## 3. Контракт модуля

Каждый модуль **обязан** экспортировать объект следующей формы:

```javascript
export const MyModule = {
  id: 'my-module',                     // уникальный идентификатор
  routes: ['/my-module', '/my-module/:id'],  // маршруты
  dependencies: ['storage', 'sync'],   // сервисы, которые нужны

  async init(ctx) {
    // Инициализация: подписки на события, загрузка данных
    ctx.events.on('entity:action', handler);
  },

  mount(container, ctx) {
    // Рендер UI в переданный DOM-контейнер
  },

  unmount() {
    // Очистка: отписки от событий, освобождение ресурсов
  }
};
```

**Правила для модуля:**
- Не использует глобальные переменные (`contractorArray`, `appSettings`, `state`, `window.*`)
- Получает всё через `ctx` (контекст с сервисами, событиями, мастер-данными)
- Общается с другими модулями **только через события** (EventBus)
- Сам управляет своим UI (рендеринг, обработчики, состояние)
- Не знает о существовании других модулей

## 4. Контекст модуля (ctx)

Контекст передаётся в каждый модуль при инициализации:

```javascript
ctx = {
  // Сервисы
  storage:     StorageService,       // IndexedDB
  sync:        SyncService,          // Supabase sync
  permissions: PermissionService,    // права доступа
  files:       FileService,          // фото, документы
  tasks:       TaskService,          // задачи
  analytics:   AnalyticsService,     // метрики

  // Мастер-данные
  masterData: {
    objects:     ObjectDirectoryService,
    contractors: ContractorDirectoryService,
    // ... другие справочники
  },

  // EventBus
  events: {
    emit(event, payload),
    on(event, handler),
    off(event, handler),
  },

  // Текущий пользователь
  user: {
    id, name, role, email
  }
};
```

## 5. Как модули обмениваются данными

**Только через EventBus.** Никаких прямых вызовов между модулями.

```javascript
// Модуль A эмитит событие
ctx.events.emit('etalon:created', { id: '123', data: {...} });

// Модуль B подписывается на событие
ctx.events.on('etalon:created', (payload) => {
  // реагирует на изменение
});
```

**Соглашение об именовании событий:** `сущность:действие`

| Событие | Значение |
|---------|----------|
| `inspection:created` | Создана новая проверка |
| `inspection:updated` | Проверка обновлена |
| `task:assigned` | Задача назначена |
| `etalon:approved` | Эталон согласован |
| `user:login` | Пользователь вошёл |
| `sync:completed` | Синхронизация завершена |

## 6. Права доступа

Permission Service централизованно управляет правами.

**Иерархия прав:** `модуль → действие → сущность`

```javascript
// Проверка права в модуле
if (ctx.permissions.can('quality', 'create', 'inspection')) {
  // показать кнопку создания проверки
}
```

**Роли системы:**

| Роль | Описание |
|------|----------|
| `admin` | Полный доступ |
| `manager` | Управление проектами, задачами |
| `inspector` | Проверки качества |
| `contractor` | Просмотр своих замечаний |
| `sk` | Стройконтроль |
| `readonly` | Только чтение |

**RLS в Supabase** дополняет проверки на уровне базы данных — двойная защита.

## 7. Подключение нового модуля (инструкция)

> Инструкция описывает **целевой порядок** после завершения platform foundation. До этого момента новые бизнес-модули не создаются (см. раздел 29).

Чтобы добавить новый модуль в платформу, нужно выполнить **5 шагов**:

### Шаг 1. Создать папку модуля
```
js/modules/my-module/
├── manifest.js           — паспорт модуля (id, title, routes, permissions, entry)
├── index.js              — единственная публичная точка входа
├── my-module.module.js   — контракт + init/mount/unmount
├── my-module.render.js   — UI-компоненты (если нужен)
├── my-module.actions.js  — бизнес-логика + вызовы сервисов (если нужен)
└── my-module.state.js    — состояние модуля (если нужен)
```

### Шаг 2. Реализовать manifest.js и index.js
`manifest.js` — метаданные, маршруты, права, поле `entry: './index.js'`.
`index.js` — реэкспортирует `init`, `mount`, `unmount` из `my-module.module.js`.

### Шаг 3. Добавить в modules.manifest.js
```javascript
// js/modules/modules.manifest.js — одна строка:
import { MyModuleManifest } from './my-module/manifest.js';
// добавить MyModuleManifest в массив ModulesManifest
```

### Шаг 4. Обновить Service Worker (если нужно)
Добавить новые файлы модуля в `urlsToCache` в `sw.js` — только если меняются кэшируемые пути.

### Шаг 5. Всё готово
`app.entry.js` не меняется. `app.js` не меняется. `index.html` не меняется.
Модуль появляется в платформе через `module-loader`, получает `ctx` через стандартный контракт.

**Принцип:** новый модуль = новые файлы в одной папке + одна строка в `modules.manifest.js`.

## 8. Пример: модуль "Согласование эталона"

Демонстрация того, как модули взаимодействуют через события:

```
┌─────────────────┐     etalon:created      ┌──────────────────┐
│  Knowledge      │ ──────────────────────► │  Approval        │
│  Module         │                         │  Module          │
│                 │                         │  (запускает      │
│  (создаёт       │                         │  воркфлоу)       │
│  эталон)        │                         └────────┬─────────┘
└─────────────────┘                                  │
                                                     │ etalon:approved
                                                     ▼
                                         ┌──────────────────────┐
                                         │  Dashboard / Quality │
                                         │  Modules             │
                                         │  (обновляют статус)  │
                                         └──────────────────────┘
```

**Воркфлоу согласования:**
1. Модуль `knowledge` создаёт эталон → эмитит `etalon:created`
2. Модуль `approval` подписан на `etalon:created` → запускает воркфлоу
3. Workflow Service создаёт задачи для ролей: генподряд, маркетинг, гарантия
4. При получении всех согласований → `approval` эмитит `etalon:approved`
5. Модули `quality` и `dashboard` реагируют → обновляют статус и UI

## 10. Как это выглядит для разработчика

**Разработчик модуля:**
- Создаёт модуль, следуя контракту (4 файла)
- Использует готовые сервисы через `ctx.*`
- Не думает об инфраструктуре (роутинг, права, синхронизация, офлайн) — это уже есть
- Может разрабатывать модуль отдельно (изолированная папка, опционально — отдельный репозиторий)

- Трогать `app.js`, `storage.js`, `sync.js`, `sw.js`
- Использовать глобальные переменные (`window.*`, `contractorArray`, `state`)
- Напрямую вызывать функции других модулей
- Менять схему IndexedDB
- Менять таблицы Supabase без добавления новых

## 11. Ключевые принципы платформы

| Принцип | Описание |
|---------|----------|
| **Модульность** | Каждый модуль независим — свои файлы, своя логика, свой UI |
| **Слабая связанность** | Модули общаются только через события, не знают друг о друге |
| **Единые мастер-данные** | Все модули используют одни справочники через `ctx.masterData` |
| **Централизованные права** | Один Permission Service — единый источник правды |
| **Offline-First** | Все данные в IndexedDB, синхронизация — опциональна |
| **Событийная архитектура** | Модули реагируют на изменения, не опрашивают другие модули |
| **Простота подключения** | Новый модуль = `manifest.js` + `index.js` + 1 строка в `modules.manifest.js`; `app.entry.js` не меняется |
| **Без регрессий** | Каждый шаг рефакторинга не ломает текущую работу приложения |

## 14. Целевая архитектура: RBI Platform Foundation

Итоговая цель текущего направления развития — не «ещё один модуль», а платформенный фундамент, на котором можно строить будущие бизнес-процессы без переписывания ядра.

**Ожидаемый результат текущего направления:**

1. Приложение продолжает работать как раньше — без регрессий.
2. Текущие функции (качество, стройконтроль, база знаний, отчёты, задачи, AI, СК, настройки) не потеряны.
3. Текущие разделы подключены к платформе как модули (через `app.entry.js`, в перспективе — через Module Manifest/Loader).
4. Wrapper используется только как временный миграционный слой, а не как постоянное решение.
5. Бизнес-логика постепенно переносится из `app.js` и legacy-файлов в модули (`state/actions/render/service`).
6. Прямые `window.*` обращения постепенно устраняются (легаси уже изолированы от инфраструктуры, следующий шаг — сам `app.js`).
7. Глобальные состояния заменяются на `state` / `services` / `ctx` / EventBus.
8. Старые legacy-функции удаляются **только после проверки зависимостей** (inline `onclick`, внешние вызовы).
9. Есть платформенный фундамент (`RBI Core` + service layer + ctx-контракт) для дальнейшего развития.
10. Новые модули можно добавлять через общую архитектуру, а не вшивать в монолит.
11. Будущие production-модули заведены только как структура/placeholder, без бизнес-логики.

## 15. Wrapper Policy / Legacy Decomposition Policy

**Wrapper — временный слой совместимости. Wrapper не является целевой архитектурой.**

Цель — не спрятать legacy под wrapper, а постепенно удалить технический долг. Каждый wrapper-модуль должен иметь план дальнейшей декомпозиции. Финальное состояние — модульная логика без зависимости от `app.js` и прямых `window.*` вызовов.

**Зачем нужен wrapper:**

- сохранить работоспособность приложения во время рефакторинга;
- безопасно подключить текущие разделы к платформенному ядру;
- не сломать текущие inline-вызовы, `window.*` прокси, sync, offline-first и PWA;
- дать возможность переносить бизнес-логику постепенно, а не одним рискованным переписыванием.

**Но wrapper не является финалом.** Каждый wrapper должен со временем «похудеть» до нуля бизнес-логики и в идеале — быть удалён, оставив только модуль (`state/actions/render/service`) + при необходимости тонкий `window.*` прокси на переходный период.

### Целевое состояние после завершения рефакторинга

- бизнес-логика вынесена из `app.js` и legacy-файлов в соответствующие модули;
- крупные legacy-файлы разобраны на `state / actions / render / service`;
- прямые глобальные обращения устранены;
- модули используют `ctx`, `services`, `EventBus`, `state` и явные зависимости;
- `window.*` прокси сохраняются только на период совместимости;
- после проверки зависимостей старые legacy-функции удаляются;
- `app.js` постепенно сокращается и перестаёт быть центром бизнес-логики;
- технический долг не маскируется wrapper-слоем, а реально устраняется.

### Полный цикл очистки для каждого крупного legacy-блока

1. Wrapper / фасад совместимости.
2. Изоляция глобальных обращений.
3. Перенос состояния в `state`.
4. Перенос действий в `actions`.
5. Перенос рендера в `render`.
6. Перенос инфраструктурной логики в `service`.
7. Замена прямых `window.*` вызовов на `ctx`/`services`/`EventBus`.
8. Проверка inline `onclick` и внешних зависимостей.
9. Удаление дублей из `app.js` / legacy.
10. Фиксация результата в `_ai/CURRENT_STEP.md`.

**Нельзя считать задачу завершённой только потому, что старая функция обёрнута через wrapper.**

Для каждого крупного блока при планировании шага нужно определять:

1. Что сейчас работает через wrapper.
2. Какая бизнес-логика остаётся внутри legacy / `app.js`.
3. Какие глобальные обращения ещё есть.
4. Какие функции можно безопасно перенести в модуль.
5. Какие `window.*` прокси пока нужно сохранить.
6. Какие зависимости нужно проверить перед удалением старого кода.
7. Какой критерий готовности считать настоящим завершением блока.

## 16. Что считать техническим долгом

- бизнес-логика внутри `app.js`;
- дубли функций между `app.js` и `*.legacy.js`;
- прямые обращения к `window.*`;
- прямые обращения к глобальным массивам и состояниям (`contractorArray`, `state`, и др.);
- прямые обращения к `STORES`, `dbPut`, `dbGet`, `triggerSync` (вне сервисного слоя);
- прямые обращения к `appSettings`;
- неявные зависимости через inline `onclick`;
- неоформленные зависимости между модулями;
- ручное подключение модулей без manifest / module loader;
- крупные legacy-файлы без внутреннего разделения (`state/actions/render/service`);
- fallback-слои, которые уже больше не нужны после стабилизации (например, `_storage()`/`_sync()` fallback на `dbPut`/`triggerSync`, когда сервис гарантированно доступен);
- сквозные сущности, хранящиеся как свободная строка вместо ссылки на каноническую запись — например, `d.contractor` в `js/construction/constructionManager.js`, который хранит отображаемое имя подрядчика (`display_name`/сырую строку из `contractorArray`), а не id/ключ из `ContractorDirectory` (см. раздел 2.4.1).

## 17. Минимальная Company

Company нужна как верхний уровень платформы и как точка будущего расширения, но **не** как тяжёлый multi-tenant engine.

**Минимальная модель:**

```javascript
Company = {
  id: 'rbi',
  name: 'RBI',
  enabledModules: [
    'quality',
    'construction',
    'knowledge',
    'reports',
    'tasks',
    'settings',
    'ai',
    'sk'
  ]
}
```

Company сейчас — простая конфигурация / context-объект, не сервис с бизнес-логикой.

**Факт реализации (2026-07-08, §29 п.11):** минимальная модель реализована как `js/services/company.service.js` (`CompanyService.getCompany()`), реальный `enabledModules` = 7 текущих platform-модулей (`quality`, `sk`, `settings`, `knowledge`, `construction`, `game`, `ai` — список в примере выше устарел, отражал более раннюю гипотезу состава модулей).

**Цель Company:**

1. Зафиксировать, что данные и доступные модули принадлежат конкретной компании.
2. Сейчас активна только компания `rbi`.
3. В будущем вторая компания должна добавляться через данные и конфигурацию, а не через переписывание ядра.

**Не создавать сейчас:**

- сложный multi-tenant engine;
- `theme.service.js`;
- `company.service.js`, если достаточно простого config/context;
- `module-access.service.js`, если достаточно `enabledModules` + `permissions`;
- переключение бизнес-логики по компании;
- модули с префиксом компании (`quality.rbi.module.js`, `quality.company-x.module.js`);
- динамическое переключение логотипов;
- динамическую смену терминологии.

Отличия будущих компаний должны решаться через `Company.enabledModules`, `permissions` по `companyId`, справочники, чек-листы, настройки, данные — не через код. Отдельный код под конкретную компанию допускается только в будущем, если появится реальная компания с реально отличающимся процессом. Сейчас это не закладывается.

## 18. Минимальная универсальная модель данных

Не строить богатую enterprise-модель заранее. Универсальность достигается не количеством полей и таблиц, а нейтральностью центральных сущностей.

**Минимальное ядро:**

- `Company` — верхний владелец данных (сейчас одна запись `'rbi'`, `enabledModules[]`);
- `Project/Object` — проект или объект;
- `LocationNode` — где: объект→корпус→секция→этаж→квартира, плоское дерево через `parentId`, не все уровни обязательны;
- `Node` — технический узел / стандарт / решение (справочник технических решений);
- `NodeApplication` — где и когда применён узел: `design`/`construction`/`operation`;
- `Assignment` — кто отвечает за локацию (подрядчик/технадзор/прораб);
- `Album` — набор стандартов / узлов компании, статус `draft`/`approved`/`etalon`;
- `File` — файл / фото / документ;
- `User / Role` — пользователь и роль.

Любая новая сущность (дефект, будущая стоимость, будущий график) ссылается на `locationId` — не создаёт свою параллельную иерархию местоположения.

**Будущие сущности не создавать сейчас** (появятся только вместе с реальными модулями, которые их используют):

- `CostItem`, `ScheduleStage`, `RiskItem`, `ContractItem`, `WorkPackage`.

> **Уточнение (2026-07-12):** `Contract` (карточка договора конкретного подрядчика — номер/дата/вид работ/статус, 1 подрядчик → N договоров) — не то же самое, что будущий `ContractItem` из списка выше (плейсхолдер для будущего модуля управления контрактами/бюджетом, раздел 2.4). `Contract` создаётся уже сейчас как прямое расширение уже существующей сущности `Contractor` (см. ниже) — минимальный набор полей договора, привязанных к подрядчику, не полноценный модуль договорного документооборота.

Обе доработки — детали в `_ai/DATA_MIGRATION_MAP.md` (секция `Contractor`), SQL — `sql/003_extend_contractors_and_contracts.sql`. Таблицы на момент этого изменения пустые/не используются приложением — не критичный шаг.

**«Пакет работ» — не обязательная центральная таблица.** Если понадобится, это представление (view) поверх существующей структуры (`LocationNode` + `Assignment` + `Node`/`NodeApplication`), не отдельная таблица `WorkPackage` в центре модели.

### 18.2 Зафиксированный будущий бизнес-сценарий: аналитика «узел × подрядчик» (2026-07-12)

> Зафиксировано архитектором по прямому запросу пользователя — чтобы не потерялось при дальнейшем планировании. Не начинать сейчас (YAGNI + зависит от незавершённых предпосылок ниже).

**Целевой сценарий:** отслеживание поведения конкретного технического узла/решения (`Node`) у конкретного подрядчика — сколько дефектов зафиксировано по этому узлу именно у этого подрядчика, повторяемость нарушений, применимость узла на разных объектах, рекомендации по узлу на основе накопленной статистики дефектов.

**Почему это невозможно технически прямо сейчас (проверено по коду 2026-07-12):**
1. `Node`/`NodeApplication` (раздел 18) — часть целевой модели данных, но **физически не существуют** ни как сущности в IndexedDB/Supabase, ни как сервис. Существующий `knowledge.service.js.getCustomNodes()`/`window.customNodes` — другая сущность (карточки базы знаний), не канонический справочник технических решений.
2. Дефекты стройконтроля (`construction_defects`) привязаны только к `floor_id`/координатам на плане (`x`/`y`), без ссылки на `Node`/`NodeApplication`.
3. Подрядчик у дефекта (`construction_defects.contractor`, `construction_acceptance.contractor`) — свободная текстовая строка, не ссылка на каноническую сущность `Contractor` (см. раздел 2.4.1, раздел 16, `_ai/DATA_MIGRATION_MAP.md` раздел «Итог по устранению технического долга §16/§2.4.1»). Без `contractorId` пересечение «узел × подрядчик» технически ненадёжно (нет канонического ключа подрядчика).

**Предпосылки, которые должны быть закрыты до начала этого сценария:**
- `Node`/`NodeApplication` созданы как реальные сущности (сервис + хранение) — сейчас есть только описание в этом документе, кода нет.
- Дефекты стройконтроля получают ссылку на `NodeApplication`/`locationId` вместо (или в дополнение к) координатам на плане.
- Завершён перевод подрядчика со свободной строки на `contractorId` — раздел 38, Блок 4 (критичный шаг, по модулям: СК → Стройконтроль → Качество).
- `construction` состоялся как полноценный второй production-модуль с достаточным объёмом реальных данных по дефектам (иначе статистика повторяемости не имеет смысла).

**Место в общей последовательности:** реалистично не раньше завершения раздела 38 Блока 4 для модуля `construction` и появления `Node`/`NodeApplication` как реальных сущностей — то есть это отдельная будущая инициатива за пределами текущего Platform Foundation (раздел 29) и её ближайшего продолжения (раздел 37/38 Блоки 2–3). Не проектировать конкретный UI/расчёт метрик заранее (YAGNI) — состав виджетов/формул определяется по факту готовности предпосылок.

### 18.1 Единый стандарт технических полей (зафиксирован 2026-07-07, реализация — раздел 38)

**Единственный и окончательный стандарт технических полей (подтверждено 2026-07-13 — «Entity Envelope» не вводится, см. §18.3).** Предметные `*Id`-поля (foreign key на другие сущности — `objectId`/`contractorId`/`nodeId`/`locationId` и т.п.) этим стандартом не покрываются и отдельной стандартизации не требуют — это обычные поля конкретной таблицы, добавляются по факту появления связи.

Каждая новая Supabase-таблица целевой схемы обязана содержать единый набор технических полей — не изобретать вариации по модулям:

```

companyId     — владелец данных (см. раздел 17, сейчас всегда 'rbi')
created_at    — момент создания
updated_at    — момент последнего изменения
deleted_at    — момент мягкого удаления (null, если не удалена)
is_deleted    — булев флаг мягкого удаления (дублирует deleted_at для быстрых фильтров)
created_by    — кто создал (userId)
version       — целочисленная версия записи (инкремент при каждом изменении, для conflict-detection синхронизации)
```

Предметные поля каждой сущности — сверх этого стандарта, специфичны для модуля. Это стандарт для **новых** таблиц целевой схемы (раздел 38). Существующие Supabase-таблицы не переименовываются под этот стандарт — они остаются как есть и переводятся в режим «только чтение» по мере миграции (раздел 38), а не приводятся к новому стандарту полей на месте.

**Цель:** текущая авторизация → Auth Gate → User Context → доступные модули → стартовый экран / default module.

**Параметризация ролевой модели по `companyId` — подпункт «(б)» пункта 10, выполнен 2026-07-13.** Целевой контракт `{ companyId, role, permissions }` реализован в `permission.service.js` (`js/services/permission.service.js`): `COMPANY_ROLE_MATRICES = { rbi: ROLE_MATRIX }` + приватный резолвер `_getRoleMatrix(companyId)` (fallback на `'rbi'` для любого неизвестного/не переданного `companyId`). 6 методов принимают `companyId` как **опциональный** параметр с fallback на текущее поведение: `getPermissions(role?, companyId?)`, `getDataScope(role?, companyId?)`, `getAllowedModules(role?, companyId?)`, `getContract(role?, companyId?)` (теперь реально использует `_getRoleMatrix`, а не хардкод `ROLE_MATRIX[r]`), `filterByDataScope(records, fieldsConfig, role?, companyId?)`, `getAllRoles(companyId?)`. Актуальное расположение техдолга синхронизации — `js/services/sync/**` (`js/sync.js` физически не существует, устаревшее упоминание удалено): вызовы `getDataScope(role)`/`filterByDataScope(...)` там уже идут через сервисный слой `window.RBI.services.permissions.*`, не bare `window.RbiRoles.*`.

**Реальное разграничение `availableModules` по роли — подпункт «(в)» пункта 10, выполнен 2026-07-13.** `permission.service.js`: legacy-константа `ALL_MODULES` (7 id: `quality`/`sk`/`settings`/`knowledge`/`construction`/`game`/`ai`, устаревшая с момента декларативной классификации `role: 'module'|'feature-of'|'service'`) заменена на `BUSINESS_MODULE_IDS = ['quality', 'construction']` — реальный список выбираемых бизнес-platform-модулей. Для всех текущих 7 ролей визуальный результат идентичен прежнему (оба модуля активны).

**Отдельно, тем же блоком или следующим за ним — параметризация ролей под организационную структуру производственного блока.** Текущие роли (`admin/manager/inspector/contractor/sk/readonly` + встречающиеся в манифестах `engineer/project_manager/deputy_manager/director`) — технические ярлыки уровня доступа, не отражающие реальную организационную структуру. ## 27. Крупные шаги вместо микрофаз

Дальнейшая разработка идёт **крупными блоками**, а не по одной функции.

**Правильные примеры:**

- Декомпозиция крупного блока `app.js`;
- Вынос крупного legacy-раздела в модуль;
- Очистка конкретного wrapper-модуля от legacy-зависимости;
- Удаление дублей после подтверждения отсутствия зависимостей;
- App Shell;
- Auth Gate + User Context;
- Minimal Company Context;
- Module Manifest + Module Loader;
- Placeholder-модули будущих разделов;
- Stabilization / Regression.

**Неправильные примеры:**

- создать одну функцию;
- переименовать одну переменную;
- заменить один вызов;
- добавить один обработчик;
- обернуть одну функцию и остановиться;
- оставить весь legacy-код под wrapper без плана удаления.

Один шаг должен быть: достаточно крупным, чтобы давать архитектурный результат; достаточно ограниченным, чтобы исполнитель мог безопасно проверить результат; не должен объединять половину платформы в один проход.

## 30. Full Application Migration

**RBI Platform — не новая оболочка рядом со старым приложением.**

RBI Platform — это целевая архитектура, в которую **полностью** переезжает текущее приложение RBI Quality Pro. Ничего не остаётся постоянным legacy-блоком внутри платформы: каждая текущая рабочая функция должна быть сохранена и подключена к платформе как одно из:

- platform module;
- internal module feature;
- service;
- shared utility;
- temporary compatibility proxy (только на переходный период, с планом удаления).

Полный переезд не означает переписывание кода или потерю функциональности — это последовательное присвоение каждому текущему runtime-файлу целевого слоя (см. `_ai/FILE_MIGRATION_MAP.md`) и постепенная физическая консолидация по плану (`_ai/APPLICATION_MIGRATION_MAP.md`).

**Критерий «полного переезда»:** ни один файл приложения не остаётся без owner module/feature и без целевого слоя; `app.js` и `index.html` перестают быть постоянными контейнерами бизнес-логики; standalone legacy JS-файлы разобраны по модулям.

## 31. Compact Module Policy

Не создавать отдельный platform module на каждую вкладку или раздел качества. **Platform module = крупное бизнес-направление.**

**Целевые platform modules текущего приложения (7):**

```
quality        — role: 'module' (главный рабочий модуль)
sk             — role: 'feature-of', parentModule: 'quality'
construction   — role: 'module' (второй рабочий модуль)
knowledge      — role: 'feature-of', parentModule: 'quality'
settings       — role: 'service' (не module — платформенный раздел настроек)
ai             — role: 'service'
gamification   — role: 'feature-of', parentModule: 'quality'
```

**Классификация `role: 'module' | 'feature-of' | 'service'` (зафиксировано 2026-07-13, блок «Декларативная классификация role»):** заменяет прежнее плоское `type: 'business' | 'platform'` (введённое 2026-07-12), которое смешивало 3 разные концептуальные роли. Актуальная модель:
1. **`module`** — самостоятельная рабочая область, выбираемая на welcome-экране: `quality`, `construction`.
2. **`feature-of: <parentModuleId>`** — внутренняя фича, принадлежащая конкретному бизнес-модулю, физически зарегистрирована в `registry` под своим ключом (историческая архитектура, не идеальная, но не меняемая этим блоком — см. `_ai/COMPACT_MODULE_RESTRUCTURE_PLAN.md`): `sk`, `game`, `knowledge` (все три — `parentModule: 'quality'`; `knowledge` при этом shared-ресурс для `construction` через `#/construction/reference`).
3. **`service`** — сквозной сервис/платформенный раздел без собственной welcome-экран рабочей области: `ai` (встроен в функции/вкладки бизнес-модулей), `settings` (платформенный раздел настроек, будущее расширение §37.2).

Поле `role`/`parentModule` пока не читается кодом (нет потребителя) — как и `type` до него, это документирующая метка для будущего разделения UI (группировка welcome-экрана/sidebar по роли). Регистр `registry`/`modules.manifest.js`/`MODULE_KEYS` этим блоком не менялся — состав и порядок 7 модулей идентичны.

**Физическое расположение (обновлено 2026-07-12, решение пользователя):** `gamification`, `ai`, `knowledge`, `sk`, `settings` физически перенесены под `js/modules/quality/features/<name>/` (не top-level папки). Это **не** слияние бизнес-логики с `quality` — каждый из 5 сохраняет собственный `manifest.js`+`index.js`+`*.module.js` и собственную регистрацию в `registry` под своим ключом (`module.game`/`module.ai`/`module.knowledge`/`module.sk`/`module.settings`). Единственное сознательное отступление от правила 1/2 Compact Module Policy ниже («internal features не получают manifest.js») — обосновано явным решением пользователя не сливать бизнес-логику 5 платформенных разделов с `quality`, только физически расположить их рядом. `construction` — единственный top-level бизнес-модуль кроме `quality`, не переносится.

**Будущие модули** (`warranty`, `budget`, `contracts`, `schedule` как отдельный бизнес-процесс, `production`) не создаются до завершения platform foundation (см. раздел 20 и раздел 33).

### Quality как один platform module с internal features

`quality` — один platform module. `audit`, `history`, `tasks`, `reports`, `analytics`, `etalon`, `engineer`, `schedule` (внутримодульный ГПР), `meetings`, `interventions` — internal features модуля `quality`, не отдельные platform modules.

```
js/modules/quality/
  manifest.js
  index.js
  quality.module.js
  quality.state.js
  quality.actions.js
  quality.render.js

  features/
    audit.js
    history.js
    tasks.js
    reports.js
    analytics.js
    etalon.js
    engineer.js
    schedule.js
    meetings.js
    interventions.js
```

Если feature слишком большая, допускается папка:

```
features/audit/
  audit.js
  audit.state.js
  audit.actions.js
  audit.render.js
```

Такие папки не создаются заранее без необходимости — только когда feature фактически большая (что уже верно для audit/history/analytics/tasks — у них исторически есть `state/actions/render`, это сохраняется при консолидации).

**Правила Compact Module Policy:**

0. **Публичная граница модуля (зафиксировано 2026-07-13):** только `index.js`/`<module>.module.js` (entry-файл модуля) имеет право писать `window.<Namespace> = ...`; файлы `features/**` — чистые ES-модули без собственных window-глобалов. Проверяется автоматически `_ai/scripts/check-module-boundaries.sh` (см. `_ai/ARCHITECTURE_BRIEF.md`, раздел «Rules»).
1. `manifest.js` + `index.js` создаются только для platform modules, не для internal features.
2. Internal features не получают собственный `manifest.js`.
3. Пустые `state`/`actions`/`render` файлы не создаются.
4. Небольшой feature может быть одним файлом.
5. Разделение на `state`/`actions`/`render` — только если feature реально большая.
6. Новый service создаётся только если логика используется несколькими модулями.
7. Legacy не переносится целиком в новые файлы без плана декомпозиции.

**Текущее переходное состояние (2026-07-05):** `js/modules/quality/*` содержит 9 отдельных `manifest.js`/`index.js` (audit, history, analytics, tasks, reports, etalon, engineer, schedule, meetings) — результат безопасной пошаговой декомпозиции `app.js` (Steps 29–39). Это **не финал**, а переходное состояние. План консолидации в один platform module — `_ai/COMPACT_MODULE_RESTRUCTURE_PLAN.md`. Текущие manifest/index не удаляются без плана и проверки загрузки.

**Целевой ориентир по количеству файлов** после очистки legacy и консолидации quality: **80–120 runtime-файлов** приложения. Переходный период может временно содержать больше файлов из-за legacy/proxy — это ожидаемо и не является нарушением политики.

## 34. index.html Migration Policy

`index.html` — не просто HTML-файл, а источник текущих зависимостей приложения (script-подключения, inline-обработчики, inline-скрипты, UI-разметка, модальные окна). Его нужно переводить в платформу по карте, поэтапно, а не одним шагом.

**План переезда `index.html`:**

1. inventory script-подключений;
2. inventory inline `onclick`/`onchange` handlers;
3. inventory inline script-блоков;
4. inventory UI-блоков;
5. inventory modal-блоков;
6. назначение owner module или app-shell для каждого элемента;
7. перенос UI в `module.render.js` / feature / app-shell;
8. удаление старых script-подключений;
9. обновление `sw.js` при необходимости;
10. smoke test;
11. финальное состояние — тонкий application shell.

Первичная инвентаризация (счётчики script-тегов, `onclick`, `onchange`) уже зафиксирована в `_ai/APPLICATION_MIGRATION_MAP.md`, раздел «index.html Decomposition Map». Детальная построчная разборка — предмет отдельного будущего блока.

**Целевое состояние:**

```html
<div id="app"></div>
<script type="module" src="./js/core/app.entry.js"></script>
```

Разбор `index.html` выполняется только поэтапно, после инвентаризации и проверок — не одним переписыванием.

## 35. app.js Decomposition Policy

`app.js` должен быть полностью разобран по модулям, features, services и shared. `app.js` не должен остаться постоянным legacy-монолитом.

Каждый крупный блок `app.js` при планировании шага декомпозиции должен получить:

- owner module;
- owner feature;
- business role;
- dependencies;
- target files;
- window proxy need;
- inline/index dependency;
- migration risk;
- removal criteria.

Текущая построчная карта оставшихся блоков `app.js` (~5764 строк на 2026-07-05) — `_ai/APPLICATION_MIGRATION_MAP.md`, раздел «app.js Decomposition Map».

**Декомпозиция идёт крупными логическими разделами, не по одной функции.**

Запрещено:

- выносить по одной строке;
- выносить по одной мелкой функции;
- оставлять весь `app.js` как постоянный legacy;
- переносить `app.js` целиком в `app.legacy.js` и считать задачу выполненной;
- переписывать половину приложения одним блоком.

## 36. Service Worker / Offline Safety

Полный переезд приложения не должен ломать: offline-first, IndexedDB, Supabase sync, PWA, Service Worker, ручное и автоматическое кэширование, локальную работу без сети.

При любом изменении путей (`manifest.js`, `index.js`, `modules.manifest.js`, `module-loader.js`, `features/*`, `module/state/actions/render` файлы, удаление standalone JS, удаление script из `index.html`) нужно проверять:

- `sw.js`;
- `cache version` (`APP_VERSION`/`SW_VERSION`, текущая 18.31.0);
- `urlsToCache`;
- offline launch;
- hard reload;
- service worker update.

Это правило действует для **каждого** блока консолидации/переноса, описанного в `COMPACT_MODULE_RESTRUCTURE_PLAN.md` и `APPLICATION_MIGRATION_MAP.md`, а не только для крупных архитектурных шагов.

## 37. Platform Entry & Module Management UI (будущий крупный блок, после Platform Foundation)

> Статус: зафиксировано как цель 2026-07-05. Не начинать до завершения пунктов 1–12 раздела 29 (Platform Foundation). В коде пока не реализовано — только описание видения.

Сейчас переключение между разделами приложения (`quality`, `construction` и т.д.) идёт через `app-mode-selector` (см. раздел 2.4.1), а вход пользователя ведёт сразу в дефолтный модуль (раздел 23, `User Context.defaultModule`). Целевая платформа должна дополнительно предусматривать два элемента, которых сейчас нет ни в коде, ни как отдельного экрана в плане:

### 37.1 Единая точка входа платформы

После Auth Gate и определения `User Context` пользователь должен попадать на экран уровня платформы (не сразу внутрь модуля), который показывает список модулей, доступных пользователю (`User Context.availableModules`, с учётом `Company.enabledModules` и прав роли). С этого экрана пользователь выбирает, с каким модулем работать дальше.

- Не заменяет `defaultModule` — оба механизма могут сосуществовать (например, автопереход в дефолтный модуль с возможностью вернуться на экран выбора).

- Является продолжением, а не заменой текущего `app-mode-selector`: `app-mode-selector` — переходный UI-элемент текущего legacy-приложения, целевой экран выбора модуля — его платформенный преемник.

**Карточки будущих модулей-заглушек (согласовано с пользователем 2026-07-12):** экран выбора модуля должен показывать не только 2 реальные рабочие карточки (`quality`, `construction`), но и карточки всех будущих направлений из раздела 20 (`safety`/`warranty`/`technical-decisions`/`contractors`/`analytics`/`schedule`/`budget`/`operation`) со статусом `planned`/`disabled`/`coming soon` — название, краткое описание, без перехода внутрь (нет `state`/`actions`/`render`, нет бизнес-логики). Реализуется через минимальный статический массив заглушек в UI-слое (`app-shell.js`/`index.html`), не через полноценный `manifest.js` на каждую заглушку — manifest создаётся только когда направление становится реальным модулем (раздел 20).

**Переприоритизация (2026-07-12, повторное решение пользователя):** §37.1 больше не отдельный блок — поглощён Блоком 2 нового плана App Shell (см. §22 ниже и §29 п.9). Welcome-экран со всеми карточками, sidebar, company block и ребрендинг "RBI Platform" делаются одним крупным блоком сразу после Блока 1 (decomposition `index.html` для `quality`).

### 37.2 Управление включёнными модулями в настройках

Модуль `settings` должен получить экран управления модулями платформы: включение/отключение модуля для компании (`Company.enabledModules`) и/или для роли (`permissions` по модулю). Сейчас `enabledModules` существует только как статический конфиг-объект (раздел 17), без UI для его изменения.

- Не создавать `module-access.service.js` заранее — управление через существующие `settings`/`permission` сервисы, пока не появится реальная необходимость в отдельном сервисе (см. YAGNI, раздел 17).
- Экран — часть модуля `settings`, не отдельный platform module.

**Целевая структура разделов внутри `settings` (зафиксировано 2026-07-12, будущее состояние, не текущий блок):** настройки качества и настройки стройконтроля должны в перспективе жить как подразделы внутри единого раздела «Настройки» платформы, не как отдельные самостоятельные экраны каждого модуля. Ориентировочные блоки: настройки платформы, настройки объектов, настройки пользователей и ролей (связано с §23), настройки справочников, настройки синхронизации, настройки качества, настройки стройконтроля, настройки будущих модулей. Это не отдельная задача сама по себе — реорганизация существующего `settings.render.js` под эту структуру начинается только когда конкретный новый блок (например §37.2 сам, либо появление настроек `construction`) реально в ней нуждается (YAGNI) — не выделяется в отдельный блок только ради реструктуризации UI.

Связано с разделом 24 (Центральный стиль) — токены `css/platform.tokens.css` должны быть управляемы (как минимум цвет/логотип компании) через настройки, без хардкода под конкретную компанию. Модули продолжают рендериться внутри `#app-content` App Shell (раздел 22), не привносят свой независимый стиль.

### 37.4 Кросс-модульный управленческий дашборд

> Статус: зафиксировано как цель 2026-07-07. Логическое продолжение раздела 37 (Platform Entry) — верхний уровень платформы для руководства, не для рядового пользователя модуля. Не начинать до появления как минимум двух-трёх production-модулей с реальными данными (сейчас есть только `quality`) — кросс-модульная аналитика физически невозможна, пока нет модулей «сроки»/«бюджет»/«подрядчики» (раздел 4.2–4.4 целевого описания платформы).

Целевой экран должен показывать руководству производственного блока сводную картину поверх всех подключённых модулей одновременно, а не по одному модулю за раз: какие объекты в риске, где критические отставания по срокам, где ухудшается качество, какие подрядчики создают наибольшие проблемы, где риск превышения бюджета, где не закрыта ИД, какие задачи просрочены, какие решения требуют участия руководства, что изменилось за неделю, какие проблемы повторяются системно.

- Строится поверх единой модели данных (раздел 18, в перспективе расширенной под `Contract`/`WorkType`/`BudgetItem`/`ScheduleItem`/`Risk`/`Decision`/`Report` — см. целевое описание платформы, раздел 5) и единой системы задач (`Task`), а не как отдельный источник данных.
- Каждый модуль публикует свои сигналы через события/сервисы (раздел 5 «Как модули обмениваются данными»), дашборд подписывается и агрегирует — не читает внутренние состояния модулей напрямую.
- Не создавать отдельный `dashboard.service.js` с бизнес-логикой заранее — экран сначала агрегирует то, что уже отдают существующие `analytics`/будущие сервисы модулей; выделять сервис только по факту реальной потребности нескольких потребителей (YAGNI, `_ai/PROJECT_RULES.md`).
- Экран — часть App Shell (раздел 22) и продолжение единой точки входа платформы (раздел 37.1), не отдельный platform module.
- Не проектировать конкретные виджеты/графики заранее — состав сигналов определяется по факту готовности модулей «сроки»/«бюджет»/«подрядчики»/«документы», не по этому описанию целиком сразу.

### Правила для этого блока

- Не начинать реализацию, пока не закрыт текущий Platform Foundation (раздел 29, пункты 1–12).
- Не проектировать сложный theme-engine или multi-tenant UI заранее — только то, что нужно для одной компании `rbi` сейчас, с заделом на `Company.enabledModules`/`permissions` (раздел 17).

## 38. Переход на новую схему Supabase-таблиц (будущая многоблочная инициатива, критичная)

> Статус: зафиксировано как план перехода 2026-07-07. **Это проектирование, не выполнение** — реализация начинается отдельными подтверждёнными блоками, не автоматически после фиксации плана. Единый стандарт технических полей — раздел 18.1.

### Последовательность (кратко, детали каждого блока — ниже)

Блок 1 — проектирование схемы/маппинга ✅ →
Блок 2 — создание новых таблиц, старые не трогаются ✅ →
Блок 3 — новые модули пишут только в новые таблицы (по факту появления первого такого модуля) →
Блок 4 — переключение записи существующих модулей на новые таблицы (критично, по одному модулю: СК → Стройконтроль → Качество; чтение старых данных сохраняется на всё время) →
Блок 5 — перенос истории старых записей в новые таблицы (по модулям, только после их Блока 4) →
Блок 6 — вывод старых таблиц из эксплуатации (отдельное решение пользователя, не автоматически).

Старые таблицы не удаляются и не теряют доступности на чтение ни на одном из шагов 1–5.

### Зачем и где это встаёт в общий рефакторинг

Это прямое следствие принципа единых данных (раздел 11, «Единые мастер-данные») и найденного технического долга (раздел 16: `d.contractor` как свободная строка вместо ссылки на каноническую сущность). Существующие Supabase-таблицы создавались раздельно под каждый раздел (осмотры качества, записи СК, дефекты стройконтроля) без единого стандарта полей и без общей сущности локации/подрядчика — что прямо противоречит целевой модели данных (раздел 18).

**Место в последовательности (раздел 29):** этот блок **не входит** в текущий Platform Foundation (пункты 1–12 раздела 29) и не блокирует его. Причины:
- Foundation сейчас работает над JS-архитектурой (модули/сервисы/ctx) поверх **существующей** схемы данных — раздел 26 прямо запрещает менять Supabase schema без отдельного решения именно для того, чтобы не смешивать два разных по риску потока изменений.
- Блок 4 этой инициативы (переключение записи существующих модулей) — критичный шаг того же уровня риска, что App Shell/Auth Gate/миграция данных (раздел `PROJECT_RULES.md`, «Критичные шаги»), и логично начинать его **после** стабилизации Foundation, когда сами модули (`quality`/`sk`/`construction`) уже физически разделены на `state/actions/render/service` — переключать источник данных проще и безопаснее в уже декомпозированном модуле, чем в монолитном legacy.
- Блок 3 (новые модули пишут только в новые таблицы) может стартовать раньше — как только появится первый новый production-модуль (например, «Гарантия», раздел 20) — то есть не раньше завершения Foundation по факту (новые бизнес-модули не начинаются до завершения Foundation, раздел 29).

Рекомендуемое место в последовательности: **Блок 1 (проектирование схемы) можно делать хоть сейчас параллельно с текущим Foundation** — это документирование, не код, риска для работающего приложения нет. Блоки 2–5 (создание таблиц, переключение записи) — после завершения пунктов 1–12 раздела 29, как один из первых пунктов раздела «Будущие этапы».

Правило без исключений: вывод старой таблицы из эксплуатации — это отдельное, вручную подтверждённое пользователем решение, принимаемое только после проверки полноты миграции истории (Блок 5). Ни один блок этой инициативы сам по себе не удаляет и не изменяет схему старых таблиц. Технически «только чтение» реализуется **на уровне приложения** (сервисный слой перестаёт вызывать `insert/update` в старую таблицу, продолжает `select`), не на уровне Supabase RLS — RLS-ограничение сочли слишком жёстким риском поломки на этом этапе.

### Блок 5 — Функция миграции истории (после Блока 4 для соответствующего модуля)

Отдельная миграционная функция (в приложении как разовый/повторяемый скрипт, либо серверная функция Supabase) читает старые записи, преобразует по маппингу Блока 1, записывает как новые записи в новые таблицы с обязательной пометкой `source_table`/`source_entity_id` (поля для прослеживаемости происхождения — не входят в стандарт раздела 18.1, специфичны только для мигрированных записей). Выполняется частями, не одномоментно — каждая порция проверяется перед продолжением. Не блокирует работу приложения во время выполнения (фоновый процесс или ручной прогон вне часов пиковой нагрузки, не синхронная блокирующая операция при пользовательском действии).

**Механизм запуска (согласовано с пользователем 2026-07-12, конкретизирует абзац выше):**

- **Кто запускает и где:** роль `admin`, через новый раздел «Миграция данных» внутри модуля `settings` (виден только роли `admin`, скрыт для остальных ролей permission-проверкой по образцу существующих admin-only блоков настроек). Не консольный/скриптовый триггер — видимый UI с прогресс-баром по каждой из 13 таблиц (сколько записей всего / перенесено / пропущено).
- **Где физически выполняется код:** в браузере администратора через уже существующий Supabase-клиент (`window.supabaseClient`, тот же `anon`/сессионный `authenticated`-ключ, что и обычная синхронизация) — не через серверную Edge Function/service_role, которых в проекте нет и добавление которых стало бы отдельным, более рискованным решением (требует Supabase CLI/Management API токена, сейчас недоступного — см. открытый технический вывод исполнителя в §38 Блок 2). Миграция — это по сути ещё один клиентский sync-сценарий: `SELECT` из старой таблицы → преобразование по карте Блока 1 → `INSERT`/`UPSERT` в новую, порциями (см. абзац выше про «не одномоментно»).
- **Записи без завершённой нормализации (`contractor_canonical_key`/`project_canonical_key` = `pending`, свободная строка вместо связи):** пропускаются при миграции, не блокируют её и не переносятся частично/с пустым `contractorId`/`locationId`. UI показывает администратору отдельный счётчик «не мигрировано, требуется ручная нормализация» с количеством и (если технически просто) ссылкой/списком таких записей. Повторный запуск миграции (идемпотентно, по уже существующему принципу `source_table`+`source_entity_id`) подтягивает эти записи автоматически после того, как их нормализация будет закрыта обычным существующим механизмом (`ContractorDirectory`/`ObjectDirectory` очередь нормализации) — отдельного специального UI «домигрировать вручную одну запись» не создаётся (YAGNI, обычный сценарий нормализации уже есть).
- **Не требовать 100% нормализации всех записей как предусловие для запуска миграции** — кнопка «Запустить миграцию» доступна всегда, когда Блок 4 для модуля закрыт; частичная миграция (с пропуском pending-записей) — ожидаемый нормальный результат первого запуска, не ошибка.

## 39. Видение: платформа как замкнутая PDCA-петля управления качеством

> **Статус: видение, не план.** Зафиксировано 2026-07-14 по прямому указанию пользователя — чисто концептуальная запись для будущего чтения при планировании этапов после Platform Foundation (13/13 §29), стабилизации `quality` на новых таблицах (§38) и стабилизации `construction`. Ничего из этого раздела не проектируется, не начинается и не превращается в блок работы, пока пользователь явно не запросит это отдельно. Точка входа для Архитектора при вопросе «что дальше» — одна строка в `_ai/ROADMAP.md`, не пересканирование этого раздела при каждом цикле.

**Platform Vision.** RBI Platform — не учёт дефектов с аналитикой сверху, а система управления жизненным циклом строительства (проектирование → эксплуатация), где ядро — замкнутый цикл PDCA вокруг версионируемого стандарта.

**Plan.** «Альбом технических решений» (тема/вид работ → ТЗ + рекомендуемые узлы + чек-лист контроля + TWI-INSPECTOR + TWI-WORKER) — версионируемая сущность. Узел имеет версию (не просто актуальную редакцию) — без версии Act не может доказать, что пересмотр стандарта снизил дефектность.

**Do.** Рейтинг подрядчика (существующий ИКО/УрК-движок) применяется не только ретроспективно («как отработал»), но и как инструмент выбора («кого назначить») — тот же расчёт, новый сценарий использования, не новая инфраструктура.

**Check.** Два независимых измерения одного чек-листа: технадзор (`construction`, поэтапная приёмка, FAIL → авто-замечание через Workflow Service подрядчику) и инженер качества (`quality`, независимая проверка результата). Расхождение оценок одного узла между ними — самостоятельный сигнал (неоднозначность стандарта либо пропуск на площадке).

**Act.** Агрегация дефектов не только по подрядчику и объекту, но и по **версии узла** (`nodeId` + `nodeVersion`) — разделяет «дефект решения» (один узел, разные подрядчики/объекты → пересмотр альбома) от «дефект исполнения» (один подрядчик на разных надёжных узлах → рейтинг подрядчика, не стандарт). Гарантийные обращения (модуль `warranty`, когда он появится) — долгий хвост той же Act-фазы, тоже ссылаются на версию узла.

**AI-ассистент сборки решений** — роль в Act-фазе, не справочный поиск: по теме (`workType`) собирает актуальную версию узла + все дефекты против неё + сравнение с предыдущей версией — прямая подготовка решения «пересматривать стандарт или нет».

**Стоимость ошибки (дни/деньги)** — лёгкое оценочное поле на дефекте/узле уровня `quality`, разрешено заполнять уже сейчас (не нарушает YAGNI — это подготовка данных, не модуль). Настоящие `schedule` (график, зависимости между видами работ, критический путь) и `budget` (согласование сумм, юридический вес, отдельная строгость доступа) — два отдельных больших модуля своей очереди (Этап 4–5 `_ai/PROJECT_CARD.md`), НЕ расширение модуля качества и не начинаются вместе с лёгким полем-оценкой.
